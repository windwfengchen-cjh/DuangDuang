### 消息转发格式规范（2026-03-25 新增 - 铁律）

**规则：子智能体在处理群内问题反馈转发时，必须使用正确的消息格式，确保@人高亮显示。**

#### 错误案例（导致@人没高亮）
子智能体直接调用 `message` 工具发送文本消息：
```json
{
  "toolCall": {
    "name": "message",
    "arguments": {
      "message": "【产研反馈-Bug】...\n<at id=\"ou_...\">施嘉科</at> ..."
    }
  }
}
```
**问题**：
- 使用了文本消息格式，不是 Post 富文本
- `<at id="...">` 是 XML 格式，飞书不识别

#### 正确做法

**方式1：使用 Post 富文本格式（推荐）**
```json
{
  "msg_type": "post",
  "content": {
    "post": {
      "zh_cn": {
        "title": "【产研反馈-Bug】",
        "content": [
          [{"tag": "text", "text": "📌 问题：..."}],
          [{"tag": "at", "user_id": "ou_...", "user_name": "施嘉科"}],
          [{"tag": "at", "user_id": "ou_...", "user_name": "宋广智"}]
        ]
      }
    }
  }
}
```

**方式2：使用 `feishu-feedback-handler` skill**
通过 skill 的转发功能，自动构造正确的 Post 格式。

#### 子智能体任务规范（必须包含）
```markdown
**消息转发规范（铁律）**：
1. 禁止直接调用 `message` 工具发送带 @ 的文本消息
2. 必须使用 Post 富文本格式：`{tag: 'at', user_id: '...', user_name: '...'}`
3. 禁止使用 `<at id="...">` 文本格式
4. 优先使用 `feishu-feedback-handler` skill 的转发功能
```

**记忆锚点**：
> "转发带@的消息 → Post格式 → `{tag:'at',user_id:'...',user_name:'...'}` → 禁止 `<at id=\"...\">`"

def should_send_severe_reminder(fields):

**错误做法：**
```python

```

**正确做法：**
```python
# ✅ 正确：使用消息原始时间
message_time = event.get('create_time', '')  # 秒级时间戳
feedback_timestamp = int(message_time) * 1000  # 转为毫秒
fields["反馈时间"] = feedback_timestamp
```

### 防御性检查
**规则：写入前必须验证时间戳合理性**

```python
def validate_feedback_timestamp(ts):
    """验证反馈时间戳"""
    if not ts or ts <= 0:
        return False, "时间戳为空或无效"
    
    now = int(time.time() * 1000)
    thirty_days_ago = now - (30 * 24 * 60 * 60 * 1000)
    
    if ts > now:
        return False, "时间戳在未来"
    if ts < thirty_days_ago:
        return False, "时间戳超过30天前"
    
    return True, "有效"
```

### 写入后验证
**规则：创建记录后必须读取验证关键字段**

record_result = record_to_bitable(...)

    if created_feedback_time is None or created_feedback_time == 0:

| 地点关键词 | 30% | 共同地点词 / 总地点词 |
| 文本相似度 | 20% | 字符交集 / 字符并集 |

### 阈值设置
- **完全重复**：相似度 > 95% → 直接返回已有记录
- **高度相似**：相似度 60%-95% → 更新原记录追加补充
- **新问题**：相似度 < 60% → 创建新记录

### 更新策略
```python
# 找到相似记录后，更新而非创建
similar_record_id = find_similar_record(content, ...)
if similar_record_id:
    update_record_with_supplement(
        record_id=similar_record_id,
        new_content=content,
        new_reporter=reporter,
        ...
    )
else:
    # 创建新记录
    create_new_record(...)
```

## 子智能体任务分配规范（2026-03-25 新增 - 铁律）

### 核心原则
**派子智能体执行任务时，必须让其使用对应的 skill，否则过程可能出错。**

### 规范说明

| 场景 | 错误做法 | 正确做法 |
|------|----------|----------|
| **需求跟进** | 派子智能体直接写代码创建群聊 | 派子智能体使用 `requirement-follow` skill |
| **PRD生成** | 让子智能体自由发挥写文档 | 派子智能体使用 `prd-document` skill |
| **表格操作** | 让子智能体用 Python 直接调 API | 优先使用 `feishu_bitable_*` 工具 |
| **消息发送** | 让子智能体构造 HTTP 请求 | 使用 `message` 工具或 skill 封装 |

### 执行策略

#### 1. 优先使用 Skill
如果任务有对应的 skill，子智能体应该：
- 读取 skill 的 `SKILL.md` 了解使用方法
- 调用 skill 的入口函数/命令
- 遵循 skill 的工作流程

**示例：**
```

→ 子智能体应使用 requirement-follow skill
→ 调用 workflow.startWorkflow() 或命令行启动
→ 不要自己重写创建群聊的逻辑
```

#### 2. 没有 Skill 时的处理
如果任务没有对应 skill：
- 使用系统提供的工具（feishu_bitable_*, message, browser 等）
- 避免直接构造 HTTP 请求（除非必要）
- 保持代码简洁，优先使用封装好的 API

#### 3. 任务描述要求
派发子智能体时，任务描述应明确：
```

❌ 不好：

```

- **子智能体调用前**：检查是否有对应 skill
- **skill 存在时**：强制使用 skill，禁止绕过
- **skill 不存在时**：记录需求，后续封装成 skill
