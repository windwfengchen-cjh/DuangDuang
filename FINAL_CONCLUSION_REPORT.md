## 根本原因分析

### 飞书权限模型（无法改变）

飞书的消息权限分为两层：

1. **API权限层**（Scope）
   - 当前应用已拥有所有必要的消息发送权限
   - `im:message:send_as_bot` ✅
   - `im:message` ✅
   - `im:message.group_msg` ✅
   - `im:message.p2p_msg:readonly` ✅

2. **可用性范围层**（Availability）
   - 这是最底层的访问控制
   - 用户必须在应用的"可用性范围"内，或主动与机器人建立过会话
   - **这是无法通过代码绕过的硬性限制**

### 关键发现

**错误码 230013**: `Bot has NO availability to this user.`
- 飞书API明确限制：机器人不能给未建立会话的用户发消息
- 这是飞书的安全设计，防止机器人滥用和骚扰用户

**错误码 232043**: `bot is invisible to user ids`
- 同样的限制，用户看不到机器人，机器人也看不到用户

## 唯一可行的解决方案

### 方案一：管理员配置可用性范围（需要管理员权限）

1. 组织管理员登录 [飞书管理后台](https://feishu.cn/admin)
2. 进入 **工作台** → **应用管理**
3. 找到应用（OpenClaw机器人）
4. 点击 **可用性设置**
5. 添加用户"梁思洁"或部门"产研-融合业务组"
6. 保存并重新发布应用

**优点**: 一劳永逸，配置后所有目标用户都能接收消息
**缺点**: 需要组织管理员权限

### 方案二：引导用户主动建立会话（无需管理员权限）

在"产研-融合业务组"群发送通知：
```
📢 系统通知

如需接收个人消息提醒，请：
1. 在飞书搜索框搜索 "OpenClaw"
2. 点击机器人头像进入私聊
3. 发送任意消息（如"你好"）
4. 完成！此后您将能接收个人通知
```

**优点**: 无需管理员权限，立即可用
**缺点**: 需要用户主动操作

## 结论

### ❌ 确认无法通过代码自动化解决

**所有技术方案均已尝试并失败。**

飞书的消息权限架构从根本上限制了机器人只能与已建立会话的用户通信。这不是API调用的技术问题，而是飞书的安全和隐私设计决策。任何声称能绕过此限制的方法都是不可行的。

**可行路径只有两条：**
1. **管理员配置**（推荐）- 一劳永逸但需要管理员权限
2. **用户主动操作** - 无需管理员权限但需要用户配合
