# auto-optimize 🔁

> 核心思想来自 [Karpathy's autoresearch](https://github.com/karpathy/autoresearch)：让 AI 在固定预算（时间/轮次）内自主迭代优化，每次实验记录结果，保留改进、丢弃倒退。

## 灵感来源

Karpathy 的 autoresearch 是一个 AI Agent 自主深度学习研究项目：
- **固定 5 分钟实验时间**（wall clock）
- **单一文件修改**（train.py）
- **只读基础设施**（prepare.py）
- **目标量化**：最小化 val_bpb（validation bits per byte）
- **实验记录**：results.tsv
- **Git 分支管理实验**

本技能借鉴其"自主迭代优化"模式，应用于更广泛的 AI 任务场景。

## 适用场景

1. **文档润色** - 持续改进文档质量，量化可读性/信息量
2. **代码重构** - 在保持功能前提下简化代码
3. **提示词优化** - 迭代改进 AI 提示词效果
4. **数据分析方案** - 比较不同分析策略的效果
5. **任何可量化评估的迭代任务**

## 工作流程

### 1. 初始化阶段

```
□ 建立 baseline（首次执行）
□ 创建结果记录文件 (results.md / results.tsv)
□ 设定评估指标（必须可量化）
□ 设定迭代预算（轮次数或时间限制）
```

### 2. 迭代循环

```
FOR i in [1..预算轮次]:
  
  1️⃣ 提出改进假设
     - 基于上一轮结果分析
     - 提出具体改进方向
  
  2️⃣ 执行修改（单一文件/任务）
     - 只修改一个方面
     - 保持变更可追溯
  
  3️⃣ 量化评估结果
     - 运行评估脚本/人工评分
     - 生成数值指标
  
  4️⃣ 记录到结果表
     - 轮次、指标、状态、改进描述
  
  5️⃣ 决策
     - ✅ keep: 指标提升 → 保留变更
     - ❌ discard: 指标下降 → 回滚
     - ⚠️  crash: 失败 → 记录原因
```

### 3. 结果记录格式

| 轮次 | 指标 | 相对提升 | 状态 | 改进描述 | 备注 |
|-----|------|---------|------|---------|------|
| 0 | baseline | - | keep | 初始版本 | - |
| 1 | +15% | +15% | keep | 增加结构化标题 | 显著提升可读性 |
| 2 | +12% | -3% | discard | 添加更多示例 | 过度冗长 |
| 3 | +18% | +6% | keep | 简化示例，突出关键信息 | 简化=胜利 |

## 设计原则（来自 autoresearch）

### 🔒 固定预算原则

- 每轮迭代有固定时间/资源预算
- 不追求单次完美，追求持续改进
- 预算内完成，超时即停
- **宁可 5 轮小改进，不要 1 轮大重构**

### 🎯 单一修改原则

- 每轮只改一个文件或一个方面
- 保持 diff 可 review
- 便于归因什么改变导致了什么结果
- **多变量同时改变 = 无法学习**

### ✨ 简化优先原则

- 相同效果，选更简单的方案
- 删除代码/内容能维持效果 = 大胜利
- 复杂改进需要显著效果才值得
- **Occam's Razor: 如无必要，勿增实体**

### 📊 量化评估原则

- 必须有可量化的评估指标
- 不接受"感觉更好"，要数字说话
- 记录所有实验，包括失败的
- **失败是数据，不是浪费**

## 使用示例

### 示例 1: 文档润色

```bash
# 初始化
任务：优化技术文档 README.md
指标：可读性评分 (1-10) + 信息密度 (关键词数/字数)
预算：5 轮迭代

# 迭代过程
轮次1: 添加目录结构 → 可读性 6→7, 保留
轮次2: 增加代码示例 → 可读性 7→7.5, 保留
轮次3: 进一步扩充示例 → 可读性 7.5→7, 丢弃（过度冗长）
轮次4: 精简核心示例 → 可读性 7.5→8, 保留 ✨
轮次5: 优化段落结构 → 可读性 8→8.2, 保留

# 结果：从 6 分提升到 8.2 分，文档更简洁有力
```

### 示例 2: 提示词优化

```bash
# 初始化
任务：优化文本分类提示词
指标：准确率 (accuracy) on 100 样本测试集
预算：10 轮迭代

# 迭代过程
轮次0: baseline prompt → 准确率 72%
轮次1: 添加 few-shot 示例 → 准确率 72%→78%, 保留
轮次2: 增加示例数量 → 准确率 78%→76%, 丢弃（过拟合）
轮次3: 优化示例选择策略 → 准确率 78%→81%, 保留
轮次4: 添加输出格式约束 → 准确率 81%→80%, 丢弃
轮次5: 简化提示词结构 → 准确率 81%→82%, 保留 ✨
...

# 结果：72% → 85%，通过 8 轮有效迭代达成
```

### 示例 3: 代码优化

```bash
# 初始化
任务：优化数据处理函数
指标：执行时间 (ms) + 代码行数 (LOC)
预算：6 轮迭代

# 迭代过程
轮次0: baseline → 120ms, 45 LOC
轮次1: 使用向量化操作 → 120ms→45ms, 保留
轮次2: 添加缓存机制 → 45ms→43ms, LOC+5, 保留（边际收益）
轮次3: 尝试并行处理 → 43ms→40ms, LOC+15, 丢弃（复杂度不值得）
轮次4: 简化缓存逻辑 → 43ms→42ms, LOC-3, 保留 ✨
轮次5: 移除冗余校验 → 42ms→41ms, LOC-5, 保留 ✨

# 结果：120ms→41ms, 45 LOC→42 LOC，更快更简洁
```

## 常见陷阱与规避

| 陷阱 | 表现 | 规避方法 |
|-----|------|---------|
| 指标选择不当 | 优化了无关指标 | 确保指标与真实目标强相关 |
| 过度拟合 | 在测试集上表现好，实际差 | 保留验证集，交叉验证 |
| 局部最优 | 陷入小改进，错过大方向 | 定期尝试激进变更 |
| 分析瘫痪 | 过度分析，行动不足 | 快速实验，数据说话 |
| 忽视失败 | 只记录成功案例 | 失败实验同样有价值 |

## 与 autoresearch 的对比

| 维度 | autoresearch (深度学习) | auto-optimize (通用技能) |
|-----|------------------------|-------------------------|
| 实验对象 | train.py | 任意可量化任务 |
| 评估方式 | 自动计算 val_bpb | 自定义评估脚本/人工评分 |
| 时间预算 | 5 分钟 wall clock | 用户自定义 |
| 版本控制 | Git 分支 | 结果表 + 可选 Git |
| 并行度 | 单实验顺序执行 | 可并行多假设测试 |

## 进阶技巧

### 1. 贝叶斯优化思路
- 记录所有实验参数和结果
- 用历史数据指导下轮假设
- 在"探索新方向"和"利用已知好方向"间平衡

### 2. 多目标优化
- 当多个指标冲突时（如速度 vs 准确率）
- 使用帕累托前沿分析
- 明确主次指标优先级

### 3. 自动化集成
- 将评估脚本化
- 使用 CI/CD 自动运行评估
- 自动生成结果报告

---

> 💡 **核心洞察**：优化是一个学习过程。每个实验——无论成败——都在教我们什么有效、什么无效。固定预算迫使我们快速行动，单一变更原则确保我们能从中学到东西。
